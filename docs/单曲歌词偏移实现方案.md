# 单曲歌词偏移实现方案

## 1. 文档目的

本文用于后续实现“播放器偏移 + 单曲偏移”中的单曲偏移部分，可直接作为另一位 Agent 的开发交接文档。

目标是允许用户针对当前歌曲修正歌词时间轴，并长期持久化。方案需要满足：

- 调整后立即生效，不重新检索歌词、不重建歌词文档。
- 播放器偏移和单曲偏移职责清晰，不出现重复补偿。
- 单曲数据可以长期增长，不污染应用设置或歌词缓存。
- 同一首歌切换播放器后仍尽量复用单曲修正。
- 托盘入口稳定，不重新引入托盘溢出面板、鼠标钩子和任务栏点击异常。

## 2. 当前实现基线

截至 2026-07-17，当前工作区已经完成播放器偏移功能。回家继续开发前先执行 `git status`，确认以下实现仍然存在；这些改动目前可能尚未提交。

| 酷狗音乐 | `+100 ms` | 提前 100 ms |

## 3. 产品定义

### 3.1 两种偏移的职责

播放器偏移用于修正播放器自身的时间轴上报延迟，例如网易云整体需要延后 130 ms。

单曲偏移用于修正某首歌的歌词时间戳，例如某份歌词整体早了 200 ms。

最终计算规则：

```text
有效偏移 = 播放器偏移 + 单曲偏移
歌词匹配位置 = SMTC 播放位置 + 有效偏移
```

示例：

```text
网易云播放器偏移：-130 ms
当前歌曲偏移：     +80 ms
最终有效偏移：      -50 ms（歌词延后 50 ms）
```

不要把单曲偏移写进歌词行时间戳，也不要修改缓存中的 `LyricDocument`。偏移只应参与运行时播放位置计算。

### 3.2 单曲偏移是否绑定播放器

第一版不绑定播放器。

原因：

- 播放器差异已经由播放器偏移负责。
- 单曲偏移描述的是歌曲/歌词时间轴修正，同一首歌跨播放器应尽量复用。
- 如果按“播放器 + 单曲”保存，会产生重复记录，用户换播放器后需要重新调整。

记录中可以保留 `SourceApp` 和 `SongId` 用于诊断，但不要把 `SourceApp` 作为第一版单曲偏移的主键组成部分。

### 3.3 数值规则

- 范围：`-5000 ms` 到 `+5000 ms`。
- 小按钮步长：`100 ms`。
- 数字输入步长：`10 ms`。
- 正值：提前。
- 负值：延后。
- `0` 不保存数据库记录，等同于删除单曲修正并继承播放器偏移。
- 播放器与单曲偏移相加后允许达到 `±10000 ms`，无需再次裁剪到 `±5000 ms`。

## 5. 持久化方案

### 5.1 使用独立 SQLite 用户数据库

推荐路径：

```text
%AppData%\TaskbarLyrics\database\user_data.db
```

不要使用以下位置：

- `settings.json`：单曲数量没有固定上限，持续增长会让设置文件越来越大，每次保存还会整体重写。
- `unified-lyrics-v8.json`：这是可清理的歌词缓存，用户偏移属于用户数据，不能随“清理歌词缓存”丢失。
- `song_maps.db`：现有数据库命名和职责是歌曲搜索映射，而且当前使用 `EnsureCreated`，直接追加表不会自动升级已经存在的数据库。

建立 `user_data.db` 可以承载未来更多用户级歌曲数据，例如偏好歌词源、纯音乐标记、歌词版本选择等。

### 5.2 表结构

建议实体名称：`TrackLyricOffset`。

```sql
CREATE TABLE TrackLyricOffsets (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    NormalizedTitle TEXT NOT NULL,
    NormalizedArtist TEXT NOT NULL,
    DurationBucketSeconds INTEGER NOT NULL DEFAULT 0,
    DisplayTitle TEXT NOT NULL,
    DisplayArtist TEXT NOT NULL,
    Album TEXT NOT NULL DEFAULT '',
    SourceApp TEXT NOT NULL DEFAULT '',
    SongId TEXT NULL,
    OffsetMilliseconds INTEGER NOT NULL,
    UpdatedAtUtc TEXT NOT NULL
);

CREATE UNIQUE INDEX IX_TrackLyricOffsets_Identity
ON TrackLyricOffsets (
    NormalizedTitle,
    NormalizedArtist,
    DurationBucketSeconds
);
```

第一版数据库是新文件，可以继续使用 EF Core `EnsureCreated`。如果 `user_data.db` 后续增加第二批表或字段，应改用 EF Core Migration，不能继续假设 `EnsureCreated` 会升级旧结构。

推荐文件：

- `TaskbarLyrics.Core/Database/UserDataDbContext.cs`
- `TaskbarLyrics.Core/Database/TrackLyricOffset.cs`
- `TaskbarLyrics.Core/Services/TrackLyricOffsetStore.cs`

实体和 DbContext 也可以暂时放在同一个文件中，避免为了很小的模型增加过多文件；存储服务应保持独立。

## 6. 歌曲身份与匹配规则

### 6.1 不直接复用当前歌词稳定身份

`LyricSyncService.BuildStableTrackIdentity` 当前只使用：

```text
SourceApp + Title + Artist
```

它的目标是防止 `SongId` 和 `Duration` 分批到达时重复检索歌词，适合当前播放会话，但不适合作为跨播放器、长期持久化的单曲偏移主键。

### 6.2 推荐规范化

为单曲偏移建立独立的 `TrackOffsetIdentity` 或 `TrackIdentityNormalizer`：

1. 标题、歌手执行 `Trim()`。
2. Unicode 规范化为 Form KC。
3. 转小写。
4. 使用现有 `ChineseScriptConverter.ToSimplified` 统一简繁体。
5. 只保留字母和数字，去除空格及标点。
6. 时长按 2 秒取整：

```csharp
(int)Math.Round(duration.TotalSeconds / 2, MidpointRounding.AwayFromZero) * 2
```

不要将专辑作为强制主键字段，因为部分播放器不提供专辑，且 SMTC 元数据会分批更新。专辑只保存用于展示和诊断。

### 6.3 查询顺序

查询当前歌曲偏移时：

1. 查找规范化标题、歌手、时长桶完全一致的记录。
2. 没有完全命中时，在相同标题和歌手下选择时长最接近的记录，最大容差建议 4 秒。
3. 仍未命中时，查找 `DurationBucketSeconds = 0` 的兜底记录。
4. 最终未命中返回 `0`。

时长未知时允许保存为 `0` 桶。后续时长稳定且用户再次调整时，可把兜底记录提升为精确时长记录。

该方案刻意不把 `SongId` 作为唯一身份，因为网易云、QQ 音乐和 Spotify 的歌曲 ID 体系不同。`SongId` 第一版只用于诊断和未来别名匹配。

### 6.4 已知取舍

- 同名、同歌手、时长非常接近的不同版本可能共享偏移，这是第一版可接受的边界。
- 不同歌词提供方可能返回时间轴不同的歌词版本。第一版仍按歌曲保存，保持用户认知简单；记录中保留歌词提供方或后续增加歌词指纹，可作为未来精确化方向。
- 不要为了第一版引入歌曲别名表、歌词版本表或多层匹配评分。

## 7. 运行时结构

### 7.1 不要每帧查询 SQLite

歌词窗口约每 60 ms 更新一次。单曲偏移解析必须走内存，不能在 `GetDisplayFrameAsync` 中访问 SQLite。

`TrackLyricOffsetStore` 建议：

- 启动时加载全部单曲偏移到内存。
- 使用 `ConcurrentDictionary`，或由内部锁保护普通字典。
- `GetOffsetMilliseconds(TrackInfo?)` 只做规范化和内存查询。
- `SetOffsetAsync` 先更新内存，使下一帧立即生效，再异步持久化。
- `RemoveOffsetAsync` 删除内存项和数据库记录。
- 数据库写入失败时记录日志并向调整窗口返回失败状态，不要让歌词线程崩溃。

一般用户的偏移记录规模远小于歌词缓存，全量加载到内存是合理的。

### 7.2 扩展 LyricSyncService

为 `LyricSyncService` 增加单曲偏移解析函数：

```csharp
Func<TrackInfo?, TimeSpan>? getTrackLeadTime = null
```

最终计算：

```csharp
var playerLead = _getPlayerLeadTime(_currentTrack?.SourceApp);
var trackLead = _getTrackLeadTime(_currentTrack);
var position = snapshot.Position + playerLead + trackLead;
```

不要在调整单曲偏移后重建 `LyricSyncService`。解析函数应读取共享的内存存储，修改后下一次计时器 Tick 自动生效。

### 7.3 App 与歌词 STA 线程

当前歌词窗口由 `LyricsWindowHost` 在独立 STA 线程中创建，托盘和设置窗口运行在 App 主 UI 线程。实现时必须显式跨 Dispatcher 获取当前歌曲。

推荐链路：

1. `MainWindow.OnTimerTick` 将 `snapshot.Track` 保存到 `_latestTrack`。
2. `MainWindow` 提供只读的当前歌曲快照。
3. `LyricsWindowHost.GetCurrentTrackAsync()` 使用 `TaskCompletionSource<TrackInfo?>` 和歌词线程 Dispatcher 返回快照。
4. `App.OpenCurrentTrackOffsetWindow` 获取快照后，在 App UI 线程打开 WPF 调整窗口。
5. `TrackLyricOffsetStore` 是线程安全的共享对象，传给 `MainWindow` 和调整窗口。

`TrackInfo` 是不可变 record，可以安全地作为快照在线程间传递。

建议由 `App` 创建并持有 `TrackLyricOffsetStore`，然后通过：

```text
App
 ├─ LyricsWindowHost
 │   └─ MainWindow / LyricSyncService（读取偏移）
 └─ TrackLyricOffsetWindow（修改偏移）
```

共享同一个存储实例。

## 9. 设置、缓存与恢复行为

- “恢复默认设置”不清除单曲偏移。
- “清除歌词与封面缓存”不清除单曲偏移。
- 如需管理全部记录，在“高级 → 数据维护”中单独增加“清除所有单曲偏移”，并要求二次确认。
- 删除某首歌偏移只删除该曲记录，不影响播放器偏移。
- 设置页播放器抽屉仍只管理播放器级配置，不在里面塞入当前歌曲配置。
- 后续可以增加单曲偏移列表、搜索和批量删除，但不属于第一版。

## 10. 诊断建议

建议在 SMTC 时间轴监视器中增加三个只读字段：

```text
PlayerOffset: -130 ms
TrackOffset:  +80 ms
Effective:    -50 ms
```

日志只在以下时机记录，不要每 60 ms 写日志：

- 当前歌曲身份发生变化。
- 命中单曲偏移。
- 用户修改或删除单曲偏移。
- SQLite 持久化失败。

建议日志示例：

```text
Track offset resolved: '歌曲 - 歌手', bucket=242, offset=80ms
Track offset updated: '歌曲 - 歌手', old=0ms, new=80ms
```
